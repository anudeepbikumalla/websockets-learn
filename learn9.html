The WebSocket protocol has built-in ping/pong frames ‚Äî but the browser JS API doesn't expose them. So we implement our own at the application level.

The Heartbeat Pattern ‚Äî client side üíª
Send a ping every 30s, wait 10s for a pong, reconnect if none
</> client.js ‚Äî heartbeat implementation
let ws, pingTimer, pongTimer;

function connect() {
  ws = new WebSocket(getWsUrl(8080));

  ws.addEventListener('open', () => {
    startHeartbeat(); // begin pinging
  });

  ws.addEventListener('message', (e) => {
    if (e.data === 'pong') {
      clearTimeout(pongTimer); // got it ‚Äî cancel timeout
    }
  });

  ws.addEventListener('close', () => {
    stopHeartbeat();
    setTimeout(connect, 3000); // reconnect
  });
}

function startHeartbeat() {
  pingTimer = setInterval(() => {
    ws.send('ping'); // send ping
    // if no pong in 10s ‚Üí connection is dead
    pongTimer = setTimeout(() => {
      ws.close(); // kill it ‚Äî close event triggers reconnect
    }, 10000);
  }, 30000); // ping every 30s
}

function stopHeartbeat() {
  clearInterval(pingTimer);
  clearTimeout(pongTimer);
}
</> server.js ‚Äî reply to ping
ws.on('message', (raw) => {
  const msg = raw.toString();
  if (msg === 'ping') {
    ws.send('pong'); // that's it!
    return;
  }
  // handle other messages normally‚Ä¶
});

Timing Guide ‚è±Ô∏è
Recommended intervals for different environments
üìã Ping interval recommendations
// General web apps (most common):
pingInterval: 30000ms // ping every 30s
pongTimeout: 10000ms // wait 10s for pong

// Mobile apps (aggressive NAT):
pingInterval: 15000ms // ping every 15s
pongTimeout: 5000ms // wait 5s for pong

// Behind AWS ALB / nginx:
pingInterval: 55000ms // just under ALB's 60s idle timeout
pongTimeout: 10000ms
Rule of thumb: set your ping interval to less than the smallest idle timeout in your infrastructure. AWS ALB = 60s ‚Üí ping every 55s.

‚ö° Heartbeat Rules
Production checklist

‚úÖ Heartbeat Checklist

üíì
Always implement heartbeats in production ‚Äî every long-lived WebSocket app needs them.
‚è∞
Stop timers on disconnect ‚Äî always call clearInterval and clearTimeout in the close handler, or you'll create ghost timers.
üîÑ
Combine with reconnect logic ‚Äî a failed pong timeout should trigger ws.close(), which then triggers your Lesson 5 reconnect logic automatically.
üìè
Keep ping payloads tiny ‚Äî just the string "ping" is fine. This is control traffic, not data.
üîï
Don't log ping/pong in production ‚Äî they fire very frequently. Filter them out of your log pipeline.

üèóÔ∏è Scenario Implementation: Stale Connection Cleanup (Heartbeat)
The Design Approach
Mobile devices moving between cell towers or losing power often drop TCP connections without sending a formal WebSocket 'close' frame. To prevent the server from running out of RAM by holding onto 'zombie' connections, the server must aggressively ping clients every ~30 seconds and terminate those that don't respond.
Implementation Code
// Run this check every 30 seconds
const interval = setInterval(() => {
  wss.clients.forEach((ws) => {
    // If client hasn't responded to the previous ping, they are dead.
    if (ws.isAlive === false) return ws.terminate();

    // Mark as false, send a new ping
    ws.isAlive = false;
    ws.ping(); 
  });
}, 30000);

wss.on('connection', (ws) => {
  ws.isAlive = true;
  // Standard WebSocket protocol specifies browsers auto-reply with 'pong'
  ws.on('pong', () => {
    ws.isAlive = true; // They responded! Mark them safe.
  });
});

üí° Key Takeaways for Beginners
‚ñ∏A heartbeat is like periodically asking "Are you still there?" ‚Äî if no response, the connection is dead.
‚ñ∏Without heartbeats, a dropped connection can stay "open" for hours ‚Äî zombie connections waste server resources.
‚ñ∏The WebSocket protocol has built-in ping/pong frames ‚Äî browsers auto-respond to pings, no client code needed!
‚ñ∏Use setInterval on the server to ping all clients every 30 seconds. Terminate anyone who doesn't respond.
‚ñ∏Load balancers (AWS ALB, Nginx) also use heartbeats ‚Äî configure their timeout to match yours.
üíª Server-side heartbeat
const wss = new WebSocket.Server({ port: 8080 });

// Every new connection starts as alive
wss.on('connection', (ws) => {
  ws.isAlive = true;  // Mark as alive

  // When client responds to ping, mark alive again
  ws.on('pong', () => {
    ws.isAlive = true;
  });
});

// Every 30 seconds, check all connections
const heartbeat = setInterval(() => {
  wss.clients.forEach((ws) => {
    if (!ws.isAlive) {
      // No pong since last check = dead connection
      console.log('üíÄ Terminating dead connection');
      return ws.terminate();  // Force close
    }

    ws.isAlive = false;  // Reset flag
    ws.ping();           // Send ping ‚Äî browser auto-responds
  });
}, 30000);  // 30 second interval

// Clean up when server shuts down
wss.on('close', () => clearInterval(heartbeat));
