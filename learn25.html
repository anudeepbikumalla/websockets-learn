Testing WebSockets üß™

WebSockets are stateful and async ‚Äî testing them requires a different approach. Learn unit tests with mocks and integration tests with a real ws client.

1
Integration Test ‚Äî real server + real client üîå
Spin up a real WS server in Jest, connect a real ws client
</> server.test.js ‚Äî Jest integration test
const { WebSocketServer } = require('ws');
const WebSocket = require('ws');

let wss, port;

beforeEach(async () => {
  wss = new WebSocketServer({ port: 0 }); // port 0 = OS picks one
  port = wss.address().port;
  wss.on('connection', (ws) => {
    ws.on('message', (msg) => ws.send('echo: ' + msg));
  });
});

afterEach(() => wss.close());

test('echoes messages', () => new Promise((resolve, reject) => {
  const ws = new WebSocket(`ws://localhost:${port}`);
  ws.on('open', () => ws.send('Hello'));
  ws.on('message', (msg) => {
    expect(msg.toString()).toBe('echo: Hello');
    ws.close(); resolve();
  });
  ws.on('error', reject);
}));
Tip: Use port: 0 to let the OS assign a random free port. This prevents port conflicts when running tests in parallel.
2
Unit Test ‚Äî mock the WebSocket üÉè
Test message handlers without a real server
</> handler.test.js ‚Äî unit test with mock
// The function to test
function handleMessage(ws, raw) {
  const msg = JSON.parse(raw);
  if (msg.type === 'ping') ws.send('pong');
}

test('responds to ping with pong', () => {
  // Mock WebSocket object
  const mockWs = { send: jest.fn(), readyState: 1 };

  handleMessage(mockWs, JSON.stringify({ type: 'ping' }));

  expect(mockWs.send).toHaveBeenCalledWith('pong');
  expect(mockWs.send).toHaveBeenCalledTimes(1);
});

Testing Disconnect & Error Scenarios üí•
Simulate close, error, and timeout events
</> Test timeout and disconnect
test('server removes user on disconnect', () => new Promise((resolve) => {
  const ws = new WebSocket(`ws://localhost:${port}`);
  ws.on('open', () => {
    expect(wss.clients.size).toBe(1);
    ws.close(); // Simulate client disconnect
  });
  ws.on('close', () => {
    setTimeout(() => { // Wait for server to process close
      expect(wss.clients.size).toBe(0);
      resolve();
    }, 100);
  });
}));

‚úÖ Testing Rules

üîå
Use port: 0 for tests ‚Äî OS assigns a random free port. Never hardcode test ports; they'll conflict in CI.
üßπ
Always close server in afterEach ‚Äî unclosed servers cause Jest to hang and timeout. Use wss.close() in cleanup.
‚è±Ô∏è
Use Promises, not done() callbacks ‚Äî returning a Promise from a Jest test is cleaner and gives better error messages on timeout.
üÉè
Unit test handlers independently ‚Äî extract message handlers into separate functions and test them with jest.fn() mock ws objects.

üí° Key Takeaways for Beginners
‚ñ∏Unit tests = mock the WebSocket (test your logic without a real server). Fast but less realistic.
‚ñ∏Integration tests = run a real server and connect real clients. Slower but catches real bugs.
‚ñ∏Use Promise wrappers to make async WebSocket tests readable: await waitForMessage(ws).
‚ñ∏Test edge cases: rapid disconnect/reconnect, malformed messages, auth failures, rate limiting.
‚ñ∏Always test the close and error paths ‚Äî they're where most production bugs hide.
