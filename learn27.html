Load Testing with Artillery üî´
Artillery is the gold standard for WebSocket load tests
üìù artillery-ws.yml ‚Äî WebSocket load test config
config:
  target: 'ws://localhost:8080'
  phases:
    - duration: 60   # 60 second test
      arrivalRate: 10 # 10 new connections/sec
    - duration: 120  # Ramp up phase
      arrivalRate: 50 # 50 new connections/sec
scenarios:
  - engine: 'ws'
    flow:
      - send: '{"type":"join","room":"load-test"}'
      - think: 5         # Stay connected 5 seconds
      - send: '{"type":"message","text":"hello"}'
      - think: 5

Run with: npx artillery run artillery-ws.yml

Quick benchmark with ws-loadtest script üíª
A simple Node.js script to create N simultaneous connections
</> loadtest.js ‚Äî quick & dirty benchmark
const WebSocket = require('ws');
const N = 1000; // connections to open
let opened = 0, failed = 0;

for (let i = 0; i < N; i++) {
  const ws = new WebSocket('ws://localhost:8080');
  ws.on('open', () => {
    opened++;
    if (opened + failed === N)
      console.log(`‚úÖ ${opened} opened, ‚ùå ${failed} failed`);
  });
  ws.on('error', () => { failed++; });
}

This quickly tells you whether your server can handle N simultaneous connections before you run a full Artillery test.

‚úÖ Load Testing Rules

üî¢
Increase OS file descriptors ‚Äî default is often 1024. Run ulimit -n 65535 before your server to allow more connections.
üìà
Test gradual ramp-up ‚Äî don't open 10,000 connections at once. Ramp up gradually (100/sec) to find the breaking point, not crash it immediately.
üéØ
Test realistic message patterns ‚Äî idle connections (no messages) are cheap. Test with realistic message rates that match your production traffic.
üìä
Monitor latency, not just connections ‚Äî 10k connections with 500ms message latency is worse than 5k connections with 5ms latency.
