const Redis = require('ioredis');
const pub = new Redis(); // for publishing
const sub = new Redis(); // for subscribing (separate connection)

// Subscribe to the shared "messages" channel
sub.subscribe('messages');

// When Redis delivers a message ‚Üí forward to ALL local WS clients
sub.on('message', (channel, data) => {
  wss.clients.forEach(client => {
    if (client.readyState === WebSocket.OPEN)
      client.send(data);
  });
});

// When a WS client sends a message ‚Üí publish to Redis
ws.on('message', (raw) => {
  pub.publish('messages', raw.toString());
  // Redis delivers it to ALL servers ‚Üí they forward to their clients
});
The magic: Every server publishes to Redis and receives from Redis. So a message from Alice (Server A) is delivered to all servers, and each server forwards it to its own local WebSocket clients.

Sticky Sessions ‚Äî an alternative (and its problems) üç¨
Route each client always to the SAME server
üç¨ Sticky Sessions

The load balancer uses the client IP or a cookie to always route the same user to the same server.

# nginx config
upstream websocket {
  ip_hash; # sticky by IP
  server localhost:8080;
  server localhost:8081;
}

‚ö†Ô∏è Problem: if Server A crashes, all its clients reconnect to B or C and lose session state.

‚úÖ Redis Pub/Sub (recommended)

No sticky sessions needed. Clients can land on any server. Redis handles message delivery between them.

// Benefits:
‚úÖ Any server can crash ‚Äî clients reconnect to any other
‚úÖ Add/remove servers without downtime
‚úÖ Works with Kubernetes autoscaling

Industry standard used by Slack, Discord, Twitch.

‚ö° Scaling Rules
Before you deploy at scale

‚úÖ Scaling Checklist

üî¥
Use Redis Pub/Sub as the message bus between your WebSocket servers ‚Äî it's the industry standard (Socket.io does this automatically).
üìä
One Node.js process can hold ~10,000‚Äì50,000 WebSocket connections before running out of file descriptors. Plan your fleet accordingly.
üîÄ
Use a load balancer with WebSocket support ‚Äî nginx, AWS ALB, or Cloudflare all support WebSocket upgrade headers.
üíæ
Never store state in memory ‚Äî room membership, user data should go in Redis, not a plain JS object, so all servers share it.
üöÄ
Try Socket.io for managed scaling ‚Äî it handles pub/sub, reconnect, and fallback automatically. Use raw ws only if you need max performance or control.
