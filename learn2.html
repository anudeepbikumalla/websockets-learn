The "Replacement" Rule

What happens when you want to do TWO things when a message arrives? Let's look at the difference between onmessage and addEventListener.

The Two Ways to Listen üéß
There are two completely different ways to tell the browser what to do.
‚ö†Ô∏è Way 1: ws.onmessage = ...
Analogy: A Single Job Title. Imagine a company that only has exactly one slot for a "Message Handler".
If you hire Alice to be the Message Handler, she does the job.

If you later hire Bob to be the Message Handler... Alice gets fired immediately. Bob overwrites her. Only one person can hold the job.
// You hire Alice
ws.onmessage = function Alice() { ... };

// You hire Bob. ALICE IS GONE!
ws.onmessage = function Bob() { ... };
‚úÖ Way 2: ws.addEventListener(...)
Analogy: A Mailing List. Imagine adding people to an email list for "Messages".
If you add Alice to the list, she gets the messages.

If you add Bob to the list... They BOTH get the messages! You just added another person to the list. Nobody gets fired.
// You add Alice to the list
ws.addEventListener('message', Alice);

// You add Bob. BOTH will run!
ws.addEventListener('message', Bob);

üí° The Golden Rule

Always use addEventListener in real apps. If you write a big app, different parts of your code might need to know when a message arrives (e.g. the chat box needs to update, AND the unread counter needs to update). If you use onmessage, they will fight and overwrite each other.

üèóÔ∏è Scenario 1: Modular Multi-Widget Dashboard (Bloomberg Terminal)
üìñ The End-to-End Story
**The Business Problem:** A financial desktop app has 15 different widgets open simultaneously: a Stock Chart, a Breaking News Feed, a Chat window with a broker, and an Account Balance display. If every widget opened its own `new WebSocket()`, the browser would choke on 15 concurrent TCP connections, draining battery and exhausting server ports.

**The End-to-End Flow:** The application architects a 'Shared Singleton Connection'. The app opens exactly ONE WebSocket to the backend. All data‚Äîstocks, news, and chat‚Äîflows down this single pipe in a massive mixed stream. Because the engineers use `addEventListener`, they can write independent, decoupled Javascript files for each widget. The News Widget attaches a listener that ignores chat messages. The Chat Widget attaches a listener that ignores stock prices. If they had used the older `ws.onmessage = handler` syntax, the last widget to load would overwrite the handlers of all the previous widgets, breaking the app.
üíª Implementation Code
// Core App Data Layer: Create ONE shared connection for the entire page
const sharedWs = new WebSocket('wss://api.trading-terminal.com/firehose');

// --- In news_widget.js ---
// This file only cares about breaking news
sharedWs.addEventListener('message', (event) => {
  const data = JSON.parse(event.data);
  // Important: We must FILTER the firehose for our specific domain
  if (data.domain === 'NEWS_ALERT') {
    document.getElementById('news-ticker').prepend(`<li>${data.headline}</li>`);
  }
});

// --- In chat_widget.js ---
// This file is completely unaware of the news widget
sharedWs.addEventListener('message', (event) => {
  const data = JSON.parse(event.data);
  // We only intercept messages meant for the chat domain
  if (data.domain === 'DIRECT_MESSAGE') {
    appendMessageBubble(data.sender, data.text);
  }
});
üèóÔ∏è Scenario 2: Global Connection Status Indicator (Slack)
üìñ The End-to-End Story
**The Business Problem:** Users in a chat app need to know if they are actually connected to the internet. If their Wi-Fi drops, they shouldn't be able to type out a 5-paragraph message only to hit 'Send' and watch it disappear into the void. They need a visual indicator (a green/red dot) at the top of the app.

**The End-to-End Flow:** Native WebSockets provide robust lifecycle events natively in the browser. The frontend engineers attach listeners directly to the formal `open` and `close` events emitted by the network layer. When the TCP handshake succeeds, `addEventListener('open')` fires, and a function paints the top-bar dot green and enables the chat input field. If the user walks into an elevator and loses signal, the TCP socket is severed, `addEventListener('close')` fires instantly, painting the dot red and disabling the text input so the user knows they are offline.
üíª Implementation Code
const ws = new WebSocket('wss://chat.enterprise.com');

// Grab our visual indicator and text input from the DOM
const statusLight = document.getElementById('connection-status');
const chatInput = document.getElementById('message-box');

// The formal TCP handshake succeeded!
ws.addEventListener('open', () => {
  statusLight.className = "indicator green";
  statusLight.innerText = "Connected";
  
  // Safe to let the user type now
  chatInput.disabled = false;
  chatInput.placeholder = "Type a message...";
});

// The formal TCP connection was severed (Wi-Fi dropped, server crashed)
ws.addEventListener('close', () => {
  statusLight.className = "indicator red";
  statusLight.innerText = "Reconnecting...";
  
  // Protect the user from losing data
  chatInput.disabled = true;
  chatInput.placeholder = "Waiting for network...";
});
üèóÔ∏è Scenario 3: Silent Analytics Tracker (Google Analytics 4)
üìñ The End-to-End Story
**The Business Problem:** A marketing team wants to track exactly where users click on the screen, how far down the page they scroll, and how long they hover over a checkout button. Sending a standard HTTP REST request for every single mouse movement would trigger massive CORS preflight overhead and DDOS the analytics server.

**The End-to-End Flow:** The application injects a silent, invisible WebSocket connection in the background. Instead of attaching a listener to the WebSocket to *receive* data, the engineers attach listeners to the *Browser DOM itself* (`document.addEventListener`). Every time the user moves their mouse or clicks, the DOM listener fires, packages the X/Y coordinates into a tiny JSON string, and pushes it up the persistent WebSocket pipe. This results in zero HTTP header overhead, allowing the tracking of thousands of micro-interactions per second without impacting page load speed.
üíª Implementation Code
// Background Tracker: No UI attached to this socket
const trackingWs = new WebSocket('wss://analytics.marketing.com/ingest');

// Listen to the user's physical mouse clicks on the webpage
document.addEventListener('click', (mouseEvent) => {
  
  // CRITICAL: We must ensure the socket is actually open before sending!
  // If we try to send data while it's still "CONNECTING" (ReadyState 0), it throws a fatal JS error.
  if (trackingWs.readyState === WebSocket.OPEN) {
    
    // Package the interaction metadata
    const payload = {
      eventType: 'USER_CLICK',
      targetTag: mouseEvent.target.tagName,
      coordinates: { x: mouseEvent.clientX, y: mouseEvent.clientY },
      timestamp: Date.now()
    };
    
    // Fire and forget via the open TCP tunnel
    trackingWs.send(JSON.stringify(payload));
  }
});

üí° Key Takeaways for Beginners
‚ñ∏ws.onmessage = fn is like writing on a whiteboard ‚Äî the next person erases what you wrote.
‚ñ∏ws.addEventListener is like a bulletin board ‚Äî everyone can pin their note and nothing gets overwritten.
‚ñ∏If two parts of your code both set ws.onmessage, only the last one works. This is a silent, hard-to-find bug.
‚ñ∏Always prefer addEventListener ‚Äî it's the modern, safe, scalable pattern.
‚ñ∏This same rule applies to all events: onopen, onclose, onerror, onmessage.
üíª Safe pattern ‚Äî use addEventListener
const ws = new WebSocket('ws://localhost:8080');

// ‚úÖ GOOD ‚Äî Multiple handlers, nothing gets overwritten
ws.addEventListener('message', (event) => {
  // Handler 1: Update the chat UI
  displayMessage(event.data);
});

ws.addEventListener('message', (event) => {
  // Handler 2: Log for debugging
  console.log('Raw message:', event.data);
});

// ‚ùå BAD ‚Äî This would OVERWRITE any previous handler:
// ws.onmessage = (event) => { ... };
// If someone else also writes ws.onmessage = ..., yours is gone!
