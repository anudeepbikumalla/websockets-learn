WebSocket Security üîí

WebSockets bypass many standard HTTP security mechanisms. An insecure WebSocket server is a wide-open attack surface. Learn the threats and how to neutralize them.

1
The 4 Main Threats üö®
Know your attack surface before you ship
üåê
Cross-Site WebSocket Hijacking
A malicious website opens a WS connection to YOUR server using the victim's cookies/session.
‚úÖ Fix: validate the Origin header on connect
üí•
DoS / Message Flooding
Attacker sends millions of messages per second to exhaust CPU and memory.
‚úÖ Fix: rate limiting (Lesson 13) + message size limit
üì®
Message Injection / XSS
Attacker sends a malicious message that gets rendered as HTML in other clients.
‚úÖ Fix: always sanitize/escape message content before rendering
üîì
No Authentication
Anyone who knows your WS URL connects and receives all messages.
‚úÖ Fix: token auth before handshake (Lesson 8)
2
Fix #1 ‚Äî Origin Header Validation üåê
Prevent cross-site WebSocket hijacking
</> server.js ‚Äî validate Origin on upgrade
const ALLOWED_ORIGINS = ['https://myapp.com', 'https://www.myapp.com'];

server.on('upgrade', (req, socket, head) => {
  const origin = req.headers.origin;

  if (!ALLOWED_ORIGINS.includes(origin)) {
    socket.write('HTTP/1.1 403 Forbidden\r\n\r\n');
    socket.destroy();
    return;
  }
  wss.handleUpgrade(req, socket, head, (ws) => {
    wss.emit('connection', ws, req);
  });
});
Important: The Origin header is set by browsers automatically and cannot be spoofed by client-side JavaScript. Server-to-server requests don't send it. So this is an effective browser CSRF protection.
3
Fix #2 ‚Äî Message Size + Content Validation üìè
Prevent DoS and injection attacks
</> server.js ‚Äî validate every message
// 1. Limit WebSocket frame size
const wss = new WebSocketServer({
  server,
  maxPayload: 64 * 1024 // 64KB max per message
});

ws.on('message', (raw) => {
  // 2. Limit message string length
  if (raw.length > 10000) { ws.close(4001); return; }

  // 3. Parse JSON safely
  let msg;
  try { msg = JSON.parse(raw); }
  catch { ws.close(4002); return; }

  // 4. Validate types (avoid prototype pollution)
  if (typeof msg.text !== 'string') return;

  // 5. Sanitize before re-sending to clients
  msg.text = escapeHtml(msg.text);
  broadcast(msg);
});

Fix #3 ‚Äî Always Use wss:// in Production üîê
Never ship ws:// in production ‚Äî TLS is mandatory
‚ùå ws:// ‚Äî unencrypted
// Anyone on the network sees
// all WS messages in plaintext!

‚úÖ wss:// ‚Äî TLS encrypted
// encrypted with TLS ‚Äî same as HTTPS

// Handle TLS in nginx/ALB, not in Node

‚úÖ Security Checklist

üåê
Validate Origin header on the HTTP upgrade event. Destroy the socket for unknown origins.
üîë
Authenticate before handshake (Lesson 8) ‚Äî verify the JWT in the query string before the connection upgrades.
üìè
Set maxPayload on WebSocketServer ‚Äî this is your first line of defense against large-payload DoS.
üßπ
Sanitize user content ‚Äî escape HTML before relaying user messages to other clients to prevent XSS.
üîê
Always use wss:// ‚Äî handle TLS termination in your reverse proxy (nginx, ALB). Never expose raw Node.js on port 443.
üåç Security ‚Äî Who Needs This?
üè¶
Banking Apps (Chase, HSBC)
All WebSocket connections use wss:// (TLS encryption). Origin checking prevents phishing sites from connecting to the bank's WebSocket server.
üí¨
Signal Messenger
Signal uses end-to-end encryption even over WebSocket. The server can't read message content ‚Äî only the sender and receiver can decrypt it.
üè•
HIPAA Healthcare Apps
Medical data over WebSocket must be encrypted (wss://), rate-limited to prevent data exfiltration, and authenticated with session tokens that expire every 15 minutes.
üîê
1Password / LastPass
Password sync between devices uses encrypted WebSocket. Input validation prevents malicious payloads from corrupting your vault.

üí° Key Takeaways for Beginners
‚ñ∏Always use wss:// (WebSocket Secure) in production ‚Äî it encrypts data like HTTPS does for HTTP.
‚ñ∏Validate the Origin header on the server ‚Äî reject connections from unknown domains to prevent CSRF attacks.
‚ñ∏Never trust client data. Always validate and sanitize every message before processing or broadcasting.
‚ñ∏Combine authentication (Lesson 8) + rate limiting (Lesson 13) + input validation = a secure WebSocket server.
‚ñ∏WebSocket connections bypass CORS ‚Äî this makes them more dangerous if not properly secured.
