Raw WebSocket messages are strings. TypeScript can't know what's inside. Learn how to use discriminated unions and Zod validation to make your WS messages fully type-safe.

Solution 1 ‚Äî Discriminated Unions ‚úÖ
Define all message shapes as a union type
</> types/messages.ts
// Define every possible message type
type ChatMessage = { type: 'chat'; text: string; user: string };
type JoinMessage = { type: 'join'; userId: string; room: string };
type LeaveMessage= { type: 'leave'; userId: string };
type TypingMessage= { type: 'typing'; username: string };

export type WSMessage = ChatMessage | JoinMessage | LeaveMessage | TypingMessage;

// Type-safe parse function
export function parseMessage(raw: string): WSMessage | null {
  try {
    const data = JSON.parse(raw);
    if (['chat','join','leave','typing'].includes(data?.type)) {
      return data as WSMessage;
    }
  } catch {}
  return null;
}
</> Using the union ‚Äî fully type-safe
ws.addEventListener('message', (e) => {
  const msg = parseMessage(e.data);
  if (!msg) return;

  switch (msg.type) { // TypeScript narrows the type!
    case 'chat': handleChat(msg.text, msg.user); break;
    case 'join': handleJoin(msg.userId, msg.room); break;
    case 'typing': handleTyping(msg.username); break;
  } // msg.tex ‚Üí TypeScript error! ‚úÖ
});
TypeScript narrows the type in each case block ‚Äî you get full autocomplete and type checking for every message shape.

Solution 2 ‚Äî Zod for runtime validation üõ°Ô∏è
Validate the actual data at runtime, not just TypeScript types
</> With Zod ‚Äî catch bad data at runtime
import { z } from 'zod';

const ChatSchema = z.object({
  type: z.literal('chat'),
  text: z.string().max(1000),
  user: z.string().min(1).max(50)
});

ws.on('message', (raw) => {
  const result = ChatSchema.safeParse(JSON.parse(raw.toString()));
  if (!result.success) {
    ws.close(4002, 'Invalid message format');
    return;
  }
  const msg = result.data; // fully typed AND validated!
  broadcast(msg);
});

Zod is runtime validation ‚Äî it catches real bad data from actual clients, not just TypeScript compile-time errors. Use both together.

‚úÖ TypeScript Rules

üü¶
Use discriminated unions ‚Äî define all message shapes with a type field. TypeScript narrows the type automatically in switch/case.
üõ°Ô∏è
Use Zod for runtime validation ‚Äî TypeScript types disappear at runtime. Zod validates real data from real (potentially malicious) clients.
üì¶
Share types between client and server ‚Äî put your WSMessage type in a shared packages/types package in a monorepo.
üöÄ
Never use as any ‚Äî if you're casting to any, you've lost all type safety. Use safeParse instead.

üèóÔ∏è Scenario Implementation: Strictly Typed Event Buses (TypeScript)
The Design Approach
When dealing with hundreds of different real-time message types, developers easily misspell JSON keys causing catastrophic frontend crashes. By defining Discriminated Union Types in TypeScript, the IDE acts as a guardrail, ensuring you only access data properties that actually exist on that specific message type.
Implementation Code
// 1. Define distinct payload shapes
interface ChatMsg { type: 'CHAT'; text: string; userId: string; }
interface MoveMsg { type: 'MOVE'; x: number; y: number; }

// 2. Combine into a Discriminated Union
type AppEvent = ChatMsg | MoveMsg;

// 3. Receive type-checked payloads
ws.onmessage = (event) => {
  // Cast raw string to our strict type
  const data = JSON.parse(event.data) as AppEvent;

  // TypeScript intelligently narrows the type based on switch!
  switch (data.type) {
    case 'CHAT':
      console.log(data.text); // Guaranteed safe
      break;
    case 'MOVE':
      console.log(`Moving to ${data.x}, ${data.y}`); // Guaranteed safe
      break;
  }
};
