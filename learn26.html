const metrics = {
  connections: 0,    // current active connections
  totalConn: 0,     // lifetime total connections
  messages: 0,     // total messages received
  errors: 0,       // total errors
  bytes: 0         // total bytes received

wss.on('connection', (ws) => {
  metrics.connections++; metrics.totalConn++;
  console.log(JSON.stringify({ event: 'connect', connections: metrics.connections }));

  ws.on('message', (msg) => { metrics.messages++; metrics.bytes += msg.length; });
  ws.on('error', () => metrics.errors++);
  ws.on('close', () => { metrics.connections--; console.log(JSON.stringify({ event: 'disconnect', connections: metrics.connections })); });
});

‚úÖ Monitoring Rules

üìù
Log as JSON ‚Äî structured logs (JSON) can be parsed by Datadog, Grafana, CloudWatch. Human-readable logs can't be queried.
üö®
Alert on connection drops ‚Äî set up an alert if active connections drop by >20% in 1 minute. Could be a server crash or deploy issue.
üì°
Expose a /metrics endpoint ‚Äî Prometheus scrapes it. Grafana visualizes it. This is the industry standard for production monitoring.
üîá
Don't log ping/pong ‚Äî they fire every 30 seconds and will flood your logs. Filter them out in your logging middleware.

üèóÔ∏è Scenario 1: AWS API Gateway + Lambda WebSockets
üìñ The Logic & Flow
**The Business Problem:** A startup builds an app that gets 0 users at night, but 50,000 active WebSocket users during a random 10-minute TV broadcast. Running massive EC2 servers 24/7 costs $5,000/month, wasting money on empty idle time.

**The End-to-End Flow:** They migrate fully to a Serverless architecture using AWS API Gateway WebSockets. Instead of a Node server holding sockets in RAM, AWS manages all 50,000 physical TCP connections automatically. When a user sends a message, AWS spins up a momentary Lambda function, passes the message to it, runs the business logic, and destroys the Lambda a microsecond later. The company now pays exactly $0.00 during the night, and only pennies per million messages during the live broadcast.
üíª Implementation Code
// Serverless Lambda Function answering an incoming WS message
const { ApiGatewayManagementApiClient, PostToConnectionCommand } = require('@aws-sdk/client-apigatewaymanagementapi');

exports.handler = async (event) => {
  // 1. AWS API Gateway passes the connection ID automatically
  const connectionId = event.requestContext.connectionId;
  const payload = JSON.parse(event.body);
  
  // 2. Process logic (e.g. save to DynamoDB)
  await db.save(payload.chatMessage);
  
  // 3. To talk BACK to the user, we issue a Command to AWS API Gateway
  const client = new ApiGatewayManagementApiClient({ 
    endpoint: `https://${event.requestContext.domainName}/${event.requestContext.stage}` 
  });
  
  const command = new PostToConnectionCommand({
    ConnectionId: connectionId,
    Data: Buffer.from(JSON.stringify({ status: "Message Saved!" }))
  });

  // Execute the command, AWS physically pushes the packet
  await client.send(command);
  
  // 4. The Lambda dies instantly. No lingering servers!
  return { statusCode: 200 };
};

üí° Key Takeaways for Beginners
‚ñ∏You can't fix what you can't see. Monitor connection count, message throughput, and error rates.
‚ñ∏Use structured JSON logs instead of plain text ‚Äî tools like Datadog and Elasticsearch can parse them.
‚ñ∏Track connection duration ‚Äî if connections are very short-lived, something is wrong (reconnect loops, auth failures).
‚ñ∏Set up alerts: connection count drops to 0, error rate spikes, message latency exceeds 1 second.
‚ñ∏Log the close code and close reason for every disconnection ‚Äî this is your debugging lifeline.
