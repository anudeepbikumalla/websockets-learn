Server-side Token Verification üíª
Verify the token in the HTTP upgrade handshake ‚Äî before the socket opens
</> server.js ‚Äî verify token on connection
const { WebSocketServer } = require('ws');
const url = require('url');

const VALID_TOKENS = new Set(['token-alice', 'token-bob']);

const wss = new WebSocketServer({ port: 8082 });

wss.on('connection', (ws, req) => {
  // Parse the URL to get the token
  const { query } = url.parse(req.url, true);
  const token = query.token;

  if (!VALID_TOKENS.has(token)) {
    // Reject with custom code 4001 = Auth failed
    ws.close(4001, 'Unauthorized');
    return;
  }

  // Token valid ‚Äî store the user identity on the socket
  ws.username = getUsernameFromToken(token);
  console.log(`‚úÖ ${ws.username} connected`);
});
Key point: Code 4001 falls in the custom range (4000‚Äì4999). On the client, check event.code === 4001 in the close handler and redirect to the login page.

‚ö° Auth Rules
Security checklist for WebSocket auth

‚úÖ Auth Checklist

üîí
Always use wss:// in production ‚Äî plain ws:// sends tokens in clear text. Use TLS (wss://) so tokens are encrypted in transit.
‚è≥
Keep tokens short-lived ‚Äî WebSocket tokens should expire (e.g. 1 hour). On expiry, close with code 4002 and force the client to refresh.
üö´
Close with custom codes on rejection ‚Äî use 4001 (Unauthorized), 4002 (Token expired), 4003 (Banned). The client can read these in the close handler.
üç™
Cookies as an alternative ‚Äî if your app uses cookie-based auth (HttpOnly session cookie), the browser sends it automatically with the WebSocket upgrade. No URL token needed!
üõ°Ô∏è
Verify server-side, not client-side ‚Äî never trust data sent in WebSocket messages for auth. A client can fake any message. Always verify at connection time on the server.

üèóÔ∏è Scenario 1: Secure Dashboard Authentication (JWT)
üìñ The Logic & Flow
**The Business Problem:** A financial dashboard cannot allow unauthenticated users to open a WebSocket and stream live banking data. However, the native `new WebSocket()` API in the browser does not support custom HTTP headers (like `Authorization: Bearer `). This breaks standard JWT authentication flows.

**The End-to-End Flow:** To bypass this browser limitation, the frontend attaches the JSON Web Token directly as a URL Query Parameter when initiating the handshake (`wss://api.com?token=xyz`). On the Node.js backend, instead of passing the raw socket to the WebSocket library immediately, the server intercepts the raw HTTP `upgrade` event. It extracts the token from the URL, cryptographically verifies the JWT, and only if valid, explicitly calls `handleUpgrade()`. If the token is fake or expired, the backend instantly destroys the TCP socket before the WebSocket protocol even starts, ensuring absolute security.
üíª Implementation Code
// Server-side: Intercepting the raw HTTP Upgrade request
server.on('upgrade', async (request, socket, head) => {
  // 1. Parse the URL to find the appended token
  const url = new URL(request.url, 'http://localhost');
  const token = url.searchParams.get('token');

  try {
    // 2. Cryptographically verify the JWT (e.g., using jsonwebtoken library)
    const user = await verifyJwt(token); 
    
    // 3. Success! Formally authorize the WebSocket handshake
    wss.handleUpgrade(request, socket, head, (ws) => {
      ws.user = user; // Attach custom user data directly to the session!
      wss.emit('connection', ws, request);
    });
  } catch (err) {
    // 4. Failure! Destroy the physical TCP connection instantly
    socket.write('HTTP/1.1 401 Unauthorized\r\n\r\n');
    socket.destroy();
  }
});
üèóÔ∏è Scenario 2: The Initial 'Hello' Auth Payload
üìñ The Logic & Flow
**The Business Problem:** Sometimes URL parameters are logged in plaintext by corporate proxies or AWS CloudTrail, making them a security risk for passing sensitive auth tokens. The application needs a way to authenticate *after* the connection opens, but *before* allowing the user to do anything else.

**The End-to-End Flow:** The server accepts all WebSocket connections blindly. However, it flags the connection state as `isAuthenticated = false`. It waits patiently. The client's very first message over the open pie must be a JSON payload containing `{ type: 'AUTH', token: '...' }`. If the client sends anything else (like a chat message), or if it takes longer than 5 seconds to send the Auth packet, the server executes `ws.close(4003)` to forcefully kick the client. Once authenticated, the server flips the boolean and allows normal traffic to flow.
üíª Implementation Code
wss.on('connection', (ws) => {
  // Lock down this specific socket
  let isAuthenticated = false;
  
  // Set a strict 5-second deadline for the client to prove who they are
  const authTimeout = setTimeout(() => ws.close(4008, 'Auth Timeout'), 5000);
  
  ws.on('message', (msg) => {
    const data = JSON.parse(msg);
    
    // Gatekeeper Logic
    if (!isAuthenticated) {
      if (data.type === 'AUTH' && isValidDbToken(data.token)) {
        isAuthenticated = true;
        clearTimeout(authTimeout); // Cancel the deadline
        ws.send(JSON.stringify({ type: 'AUTH_SUCCESS' }));
      } else {
        ws.close(4003, 'Invalid Token');
      }
      return; // Stop execution!
    }
    
    // Process normal application messages here...
  });
});

üí° Key Takeaways for Beginners
‚ñ∏WebSockets can't send custom HTTP headers during the handshake ‚Äî so we pass the token in the URL query string.
‚ñ∏Always validate the token on the server immediately during the connection event. Reject bad tokens instantly.
‚ñ∏Custom close codes 4000-4999 are yours to define ‚Äî use 4001 for unauthorized, 4002 for expired token, etc.
‚ñ∏JWTs (JSON Web Tokens) are the most common auth method ‚Äî they contain user data encoded inside the token itself.
‚ñ∏Never trust the client ‚Äî always verify server-side. A malicious user can send any token they want.

// CLIENT ‚Äî Connect with auth token in URL

const ws = new WebSocket(getWsUrl(8082) + `?token=${token}`);

// SERVER ‚Äî Validate the token on connection

  // 1. Extract token from the URL query string
  const url = new URL(req.url, 'http://localhost');
  const token = url.searchParams.get('token');

  // 2. Verify the token (using JWT or your own logic)
  try {
    const user = jwt.verify(token, SECRET_KEY);
    ws.userId = user.id;  // Attach user info to connection
    ws.send(JSON.stringify({ type: 'auth', status: 'ok' }));
  } catch (err) {
    // 3. Reject invalid tokens immediately!
    ws.close(4001, 'Invalid or expired token');
    return; // Stop processing
  }

  ws.on('message', (msg) => {
    console.log(`Message from user ${ws.userId}:`, msg);
  });
});
